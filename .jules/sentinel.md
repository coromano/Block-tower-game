## 2024-03-21 - [Missing Treasury Address Constraint on CHECK Accounts]
**Vulnerability:** The `tesoreria` account in the `ComprarBloque` instruction lacked an explicit `address` constraint. It only used `/// CHECK: Tesorería del juego` without verifying if the provided account matched the actual expected treasury pubkey. This allowed an attacker to pass an arbitrary account (e.g. their own wallet) as the `tesoreria` account, effectively receiving blocks without paying the real treasury.
**Learning:** In Solana/Anchor, `/// CHECK:` is insufficient for critical accounts like fund destinations. When transferring funds to a designated treasury or fee receiver, the account must be strictly checked to match the expected hardcoded or programmatic public key, otherwise an attacker can spoof the destination.
**Prevention:** Always use the `address` constraint (`#[account(mut, address = expected_pubkey)]`) or an explicit `require_keys_eq!` check inside the instruction handler for accounts that receive protocol fees or treasury funds.
